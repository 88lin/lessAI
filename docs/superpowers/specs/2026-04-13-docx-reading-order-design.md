# Docx Reading Order Design

## Goal

在不追求 Word 像素级排版一致的前提下，将 `.docx` 导入结果提升到“阅读顺序尽量一致”。

## Confirmed Constraints

- 仅面向文章语义相关文档。
- 非文字内容不解析正文语义，统一显示为锁定占位符。
- 可被 AI 改写的内容必须可安全写回。
- 不支持的复杂结构直接显式报错，不做 silent fallback。

## Problem

当前 `docx` 导入主要按 `document.xml` 事件流线性展开。该顺序不等于 Word 的阅读顺序，尤其在以下场景会失真：

- 自动编号存在于 `w:numPr + numbering.xml`，不是正文字符。
- 填写线常是“带下划线的空白 run”，不是图形对象。
- 浮动文本框/图形锚定在某段，但视觉上位于其他位置。

## Design

### 1. 阅读顺序模型

导入阶段先抽取段落内语义片段，再对“可见但不可改写”的结构做阅读顺序重排。目标是构建稳定的阅读流，而不是复刻版面引擎。

### 2. 自动编号

解析 `word/numbering.xml`，将常见编号前缀渲染为锁定前缀，例如 `1、`。该前缀参与阅读，但不进入可改写正文。

### 3. 填写线

识别“仅空白字符 + 下划线样式”的 run，并显示为锁定占位符 `[填写线]`。写回继续保留原始 run，不允许编辑器改写占位符内容。

### 4. 浮动对象顺序

对文本框等浮动对象提取锚点元数据与近似纵向位置，不再按 XML 出现位置直接内联插入；改为独立锁定块，并尽量排到更接近阅读位置的段落之后。

### 5. 边界

不做完整布局，不承诺页内像素级一致；仅提升阅读顺序、结构感知和占位稳定性。

## Validation

- 模板文档中能看到自动编号、填写线、文本框占位。
- `[文本框]` 不再错误地夹进日期句中。
- 导入后仍保持“可改写 == 可写回”。
